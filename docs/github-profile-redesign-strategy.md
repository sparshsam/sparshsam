# GitHub Profile Redesign Strategy

## Purpose

The GitHub profile should function as a research index, not a portfolio wall. It should help readers quickly understand:

- what problem space the ecosystem studies;
- which repositories belong to which research area;
- how to cite or archive serious outputs;
- how the work differs from ordinary product demos;
- how to contribute without changing the tone of the ecosystem.

## Profile Structure

1. Identity line: independent researcher, open-source systems designer, behavioral infrastructure builder.
2. Research direction: calm software, local-first computing, proof systems, ownership without surveillance.
3. Project table: one row per public repository, grouped by research area.
4. Current outputs: links to narrative, philosophy, roadmap, taxonomy, ORCID/DOI workflow.
5. Maintenance principles: concise rules that signal seriousness.
6. ORCID line: canonical public iD.

## Tone Rules

- Use infrastructural language.
- Avoid hype terms such as revolutionizing, disrupting, moonshot, next-generation, or web3-native.
- Avoid crypto-forward framing. When chain infrastructure is relevant, describe it factually as a verification or custody substrate.
- Avoid exaggerated claims. Prefer inspectable claims: local-first, exportable, self-hostable, timestamped, archived, citeable.
- Keep interface philosophy visible, but subordinate to engineering substance.

## Profile Maintenance Rhythm

- Monthly: update project descriptions, research outputs, and roadmap status.
- Per release: add DOI, citation, and changelog links.
- Quarterly: review pinned repositories and consistency of repo topics.
- Annually: publish a research summary, architecture paper, or ecosystem report.
